# ТЗ: анимированный персонаж (лемминг) — для внешнего подрядчика (LLM)

Версия 1.0 · Движок: **Godot 4.6 (GDScript)** · Платформы: **iOS + Android** (приоритет), Linux/Mac/Windows (dev)

Этот документ — единственный источник правды по технической части. Цель — получить
**анимированного, легко дорисовываемого персонажа**, который встраивается в текущую игру
**без переделки игровой логики**. Любое отклонение от раздела «Жёсткие инварианты» делает
поставку непринимаемой.

> **Статус реализации (2026-06-13).** Первый персонаж сделан **растром**, а не SVG-ригом:
> внешняя нейросеть нарисовала листы поз (`assets/characters/lemming/ref/`),
> `tools/slice_character.py` режет их по связным альфа-компонентам и нормализует
> (единый масштаб, стопы на базовой линии), а рантайм-адаптер
> `entities/lemming_sprite.gd` играет результат через `AnimatedSprite2D`, сохраняя все
> «жёсткие инварианты» ниже. Жёсткие инварианты (§2) и приёмка (§10) актуальны для любого
> формата. SVG-риг-конвейер (§7–§8) остаётся целевым для будущего: он даёт покадровую
> консистентность и плавные циклы, которых растровый AI-арт не обеспечивает.

---

## 0. TL;DR для подрядчика

Ты сдаёшь **векторного персонажа в виде SVG-рига + данных анимации (JSON) + инструмента
запекания**. Конвейер:

```
SVG-части тела  +  JSON-кейфреймы по состояниям
            │  (bake-инструмент, headless Godot)
            ▼
   SpriteFrames (.tres) + атлас-текстура(ы)
            │  (рантайм-адаптер, тот же API)
            ▼
   AnimatedSprite2D внутри узла "Sprite"
```

**Почему так, а не «живой SVG»:** Godot растеризует SVG при импорте в текстуру и НЕ
анимирует вектор в рантайме. А в кадре на телефоне бывает **до ~100 леммингов
одновременно**, поэтому скелетный риг на каждом юните недопустим по производительности.
Решение: вектор — это *исходник* (его бесконечно дорисовывают и масштабируют), а в игру
едут **запечённые кадры** в общий `SpriteFrames` — дёшево и детерминированно.

## 1. Что именно сдаётся (Deliverables)

| # | Артефакт | Путь в репозитории | Формат |
|---|---|---|---|
| 1 | Библиотека SVG-частей тела (риг) | `assets/characters/lemming/parts/*.svg` | SVG 1.1, по одной части на файл |
| 2 | Описание рига и кейфреймов | `assets/characters/lemming/rig.json` | JSON (схема в §7) |
| 3 | Инструмент запекания | `tools/bake_character.gd` | GDScript (`EditorScript` или headless-скрипт) |
| 4 | Запечённый результат | `assets/characters/lemming/lemming_frames.tres` (+ `.png` атлас(ы)) | `SpriteFrames` |
| 5 | Рантайм-адаптер | `entities/lemming_sprite.gd` (замена существующего) | GDScript, `class_name LemmingSprite` |
| 6 | Автотесты | `tests/test_lemming_sprite.gd` | GUT |
| 7 | README по конвейеру | `assets/characters/lemming/README.md` | Markdown: как добавить деталь и пере-запечь |

Артефакты 1–3 — «исходники» (то, что правит человек/LLM). Артефакты 4–5 — то, что реально
исполняется в игре. **Запекание должно быть воспроизводимым**: повторный прогон bake-инструмента
из тех же исходников даёт идентичный результат.

## 2. Жёсткие инварианты интеграции (НАРУШАТЬ НЕЛЬЗЯ)

Эти точки уже зашиты в игровую логику (`entities/lemming.gd`, меню, карта мира). Адаптер
обязан их сохранить 1:1.

1. **Узел и базовый класс.** В сцене `entities/lemming.tscn` дочерний узел называется
   `Sprite`, имеет скрипт `entities/lemming_sprite.gd` с `class_name LemmingSprite` и
   **наследуется от `Node2D`**. (Внутри узла можно держать `AnimatedSprite2D` как ребёнка —
   см. §8.) Узел `Sprite` расположен в родителе на позиции `(8, 16)`.

2. **Якорь = ноги в локальном `(0, 0)`.** Точка `(0,0)` адаптера — это пиксель, на котором
   лемминг стоит (центр нижней грани тела). Тело уходит вверх в **отрицательный y**. Низ
   стопы при любой позе обязан совпадать с `y=0 ± 1px` (кроме `splat` — он лежит на земле,
   тоже от `y=0` вверх). Это критично: физика ставит ноги по маске террейна, визуал обязан
   садиться ровно.

3. **Публичный API адаптера — ровно два свойства:**
   ```gdscript
   var state: int   # значение из Lemming.State (см. §4); сеттер меняет анимацию
   var dir: int     # 1 = вправо (как нарисовано), -1 = зеркало по горизонтали
   ```
   Любой код игры выставляет только их (`sprite.state = ...`, `sprite.dir = ...`). Менять
   сигнатуру нельзя.

4. **`dir = -1` — это горизонтальное зеркало**, без сдвига позиции. Исходник рисуется
   «лицом вправо»; влево получается отражением (`flip_h` / `scale.x = -1`).

5. **`modulate` зарезервирован движком.** Родитель сам красит узел: подсветка кандидата
   `Color(1.7, 1.7, 0.7)`, мигание бомбы между `Color(1,0.4,0.4)` и `Color(1,1,0.4)`.
   Адаптер **не должен трогать `self.modulate`** (анимируй цветом только внутренние слои,
   если нужно, но не корневой модулят).

6. **Анимация — только визуальные часы по `delta`, НИКОГДА по sim-тику.** Физика лемминга
   детерминирована и используется в реплеях; визуал обязан быть полностью отвязан. Кадр
   выбирается из внутреннего счётчика, накапливаемого в `_process(delta)`. Визуальный
   рассинхрон не имеет права влиять на симуляцию.

7. **Размер коллизии НЕ меняется.** `CollisionShape2D` (капсула r=5, h=13, на `(8, 9.5)`) и
   логический «footprint» 16×16 px остаются как есть — на них завязана пиксельная физика.
   Визуал может выходить за коллизию (конечности, зонт), но «стопа» обязана садиться по §2.2.

8. **Работа без родителя-лемминга.** Адаптер используется и в меню (марширующие лемминги),
   и на карте мира (аватар) как самостоятельный `LemmingSprite` без `Lemming`-родителя.
   Он обязан корректно рисоваться только из `state` + `dir`, ничего не зная об игре.

9. **Фильтр текстур.** По умолчанию игра — пиксель-арт (`TEXTURE_FILTER_NEAREST`, террейн
   рисуется шейдером). Согласуй вид: либо запекай в «родном» масштабе под NEAREST, либо
   запекай ×N с мип-мапами под линейную фильтрацию — но итог должен визуально сочетаться с
   пиксельным террейном (см. §6, это пункт арт-ревью).

## 3. Контекст: как персонаж используется в игре

- `entities/lemming.gd` хранит `@onready var sprite: LemmingSprite` и в `_update_visual()`
  выставляет `sprite.dir` и `sprite.state`. Больше ничего о визуале он не знает.
- Состояние и направление меняются десятки раз за прохождение; переходы должны быть мгновенными
  и без артефактов (не «доигрывать» предыдущую анимацию).
- Лемминги создаются и удаляются (`queue_free`) по ходу уровня; ресурс `SpriteFrames` —
  **общий** (один на всех, грузится один раз), текстура атласа в VRAM в единственном экземпляре.

## 4. Состояния и анимации

`Lemming.State` (порядок = целочисленные значения 0..12):

| int | enum | Имя анимации (clip) | Луп | Что показываем |
|----|------|---------------------|-----|----------------|
| 0  | WALKING   | `walk`   | да  | Спокойная ходьба, мах руками/ногами, чуб колышется |
| 1  | FALLING   | `fall`   | да  | Падение: руки вверх, тело напряжено |
| 2  | FLOATING  | `float`  | да  | Раскрытый зонт сверху, лёгкое покачивание |
| 3  | CLIMBING  | `climb`  | да  | Лезет по стене: тело прижато, руки перехватывают вверх |
| 4  | BLOCKING  | `block`  | да  | Стоит, обе руки в стороны «стоп», устойчивая стойка |
| 5  | BUILDING  | `build`  | да  | Наклон вперёд, кладёт планку, ритмичное движение руки |
| 6  | BASHING   | `bash`   | да  | Горизонтальные удары рукой/инструментом вперёд |
| 7  | MINING    | `mine`   | да  | Удар киркой вниз-вперёд под 45° |
| 8  | DIGGING   | `dig`    | да  | Копает вниз: тело покачивается, руки поочерёдно гребут |
| 9  | EXPLODING | `panic`  | да  | «Oh no!»: руки вверх, дрожь (тинт мигает — делает родитель) |
| 10 | DYING     | `splat`  | нет | Смерть (общая с splat) |
| 11 | EXITED    | `cheer`  | нет | Радость на выходе: руки вверх (юнит сразу удаляется, 1–2 кадра ок) |
| 12 | SPLAT     | `splat`  | нет | Расплющен в «блин», чуб торчит |

Требования:
- Каждое из 13 значений `state` обязано отображаться (DYING и SPLAT делят `splat`).
- Циклические клипы (`walk`, `dig`, `bash`, …) — бесшовный луп.
- `splat`/`cheer` — однократные (`loop = false`), последний кадр «замирает».
- **Скорость анимаций** ориентируй на текущие частоты (для совпадения с ощущением физики):
  ходьба/бег рук ≈ цикл 0.55 с; копание/удары ≈ цикл 0.45–0.7 с; зонт качается ≈ 2 с;
  паника-дрожь — быстрая. Точные fps/число кадров — на твоё усмотрение в рамках §6.

Референс поз (как сейчас отрисовано процедурно) — `entities/lemming_sprite.gd`, методы
`_draw_<state>()`. Используй их как функциональный ориентир по позам и читаемости силуэта, но
визуал делай **детальнее и пластичнее**.

## 5. Стиль и узнаваемость

- Сохрани узнаваемый силуэт текущего юнита: **зелёный чуб**, **синяя роба**, светлая кожа,
  оранжевые ботинки (палитра — константы в `entities/lemming_sprite.gd`: `HAIR`, `SKIN`,
  `ROBE`, `BOOT`, `EYE`). Можешь обогатить шейдингом/оттенками, но юнит должен «читаться» как
  тот же персонаж.
- Силуэт обязан быть **читаемым на ~16–20px высоте** и при 50+ юнитах в кадре: крупная
  голова, контрастная роба, отличимые позы по состояниям (издалека видно, копает он или строит).
- Палитру держи в SVG как именованные слои/группы, чтобы перекраску можно было сделать правкой
  нескольких значений (на будущее — варианты юнитов/команд).
- Юридическое замечание: финальный бренд (имя/арт) ещё не утверждён (бэклог US-5.4, deferred).
  Не копируй буквально арт оригинальной Lemmings; держись текущего собственного образа.

## 6. Геометрия, холст кадра, разрешение

- **Footprint / коллизия:** 16×16 px — НЕ менять (см. §2.7).
- **Логический холст кадра:** ширина 32 px, высота 40 px (с запасом на зонт во `float` и
  поднятые руки в `panic`/`cheer`). Персонаж центрирован по горизонтали; **низ стопы — на
  отметке, соответствующей локальному `(0,0)` адаптера** (рассчитай `offset`/`region` так,
  чтобы это выполнялось — это проверяется тестом, §10).
- **Номинальная высота тела** (стопа→макушка) ≈ 18–22 px. Конечности/зонт могут выходить выше.
- **Супер-сэмплинг для чёткости и Retina:** авторь и запекай в ×4 (исходный холст 128×160 px),
  даунскейл в атлас — это даёт чистые края на высоких DPI. Финальный размер отрисовки в игре —
  логические 32×40 px.
- **Pivot/offset:** так, чтобы выполнялся §2.2. Для `AnimatedSprite2D` это `centered=false` +
  точный `offset`, либо `centered=true` + `offset` с поправкой. Зафиксируй и задокументируй.

## 7. Формат исходников

### 7.1 SVG-части (`parts/*.svg`)

- SVG 1.1, **только векторные пути/примитивы**: `<path>`, `<polygon>`, `<circle>`, `<rect>`,
  `<g>`. Запрещено: растровые `<image>`, внешние шрифты/`<text>`, скрипты, внешние ссылки,
  фильтры размытия/тени (`feGaussianBlur` и т.п. — Godot их игнорирует/искажает), CSS-классы из
  внешних файлов.
- Цвета — инлайн (`fill="#RRGGBB"`), сетка координат в тех же единицах, что и холст части.
- Один файл = одна логическая часть (например: `head.svg`, `hair.svg`, `torso.svg`,
  `arm_front.svg`, `arm_back.svg`, `leg_front.svg`, `leg_back.svg`, `boot.svg`, `umbrella.svg`,
  `pick.svg`, …). Точка вращения части (pivot) задаётся в `rig.json`, не в SVG.

### 7.2 Риг и кейфреймы (`rig.json`)

Декларативное описание: список частей с пивотами и порядком отрисовки (z) + по одной анимации
на каждое имя клипа из §4, с кадрами-ключами и трансформами частей. Пример схемы:

```jsonc
{
  "canvas": { "w": 32, "h": 40, "feet_x": 16, "feet_y": 40 },  // feet_* в px холста
  "supersample": 4,
  "parts": [
    { "id": "leg_back",  "svg": "parts/leg_back.svg",  "pivot": [4, 2],  "z": 0 },
    { "id": "torso",     "svg": "parts/torso.svg",     "pivot": [6, 14], "z": 2 },
    { "id": "head",      "svg": "parts/head.svg",      "pivot": [5, 8],  "z": 4 }
    // ...
  ],
  "clips": {
    "walk": {
      "fps": 12,
      "loop": true,
      "frames": [
        { "leg_front": { "pos": [16, 26], "rot": 8,  "scale": [1,1] },
          "leg_back":  { "pos": [16, 26], "rot": -8, "scale": [1,1] },
          "torso":     { "pos": [16, 24], "rot": 0 },
          "head":      { "pos": [17, 16], "rot": 1 }
          /* части, не упомянутые в кадре, берут трансформ из "rest"/предыдущего */ },
        { /* кадр 2 ... */ }
      ]
    },
    "dig":   { "fps": 10, "loop": true,  "frames": [ /* ... */ ] },
    "splat": { "fps": 8,  "loop": false, "frames": [ /* ... */ ] }
    // ... все клипы из §4
  }
}
```

- Координаты — в px логического холста (до супер-сэмплинга); `rot` в градусах; `scale`
  опционален (по умолчанию `[1,1]`).
- Схема может быть расширена (easing между ключами, отдельный `rest`-pose и т.п.) — это на твоё
  усмотрение, но она обязана быть **самодостаточной и детерминированной**: bake-инструмент
  строит из неё кадры без ручных шагов.

## 8. Bake-инструмент (`tools/bake_character.gd`)

- Запуск headless: `godot --headless -s tools/bake_character.gd` (и/или как `EditorScript`).
- Вход: `assets/characters/lemming/parts/*.svg` + `rig.json`.
- Действие: для каждого клипа и кадра — собрать части по трансформам, отрисовать в
  `SubViewport`/`Image` размера `canvas × supersample`, даунскейлить до `canvas`, сложить кадры
  в атлас и сформировать `SpriteFrames` с анимациями, **имена которых = имена клипов из §4**,
  с корректными `loop` и `fps`.
- Выход: `assets/characters/lemming/lemming_frames.tres` + атлас-текстура(ы) `.png` с `.import`.
- **Детерминизм:** без `Time`/`randf()`; повторный прогон из тех же исходников даёт
  побайтово одинаковый результат (это упрощает ревью диффов).
- Скрипт логирует, что собрал (сколько клипов/кадров), и падает с понятной ошибкой, если в
  `rig.json` не хватает клипа из §4.

> Альтернатива для простоты, если рантайм-сборка частей в `SubViewport` окажется слишком
> громоздкой: допускается сдать по одному собранному SVG-кадру на позу и запекать их через
> штатный SVG-импорт Godot. Но риг-подход (части + кейфреймы) предпочтителен — он убирает
> межкадровый «джиттер» и максимально упрощает «дорисовывать».

## 9. Рантайм-адаптер (`entities/lemming_sprite.gd`)

Заменяет текущий процедурный скрипт, сохраняя контракт §2.

```gdscript
class_name LemmingSprite
extends Node2D

# Внутри — AnimatedSprite2D с ОБЩИМ ресурсом SpriteFrames (грузится один раз).
# Публичный API ровно тот же: state:int (Lemming.State) и dir:int (1/-1).

const FRAMES := preload("res://assets/characters/lemming/lemming_frames.tres")
const STATE_TO_CLIP := {
    0: "walk", 1: "fall", 2: "float", 3: "climb", 4: "block",
    5: "build", 6: "bash", 7: "mine", 8: "dig", 9: "panic",
    10: "splat", 11: "cheer", 12: "splat",
}

var state: int = 0:
    set(v): state = v; _apply()
var dir: int = 1:
    set(v): dir = v; _apply()
# ... _anim: AnimatedSprite2D; _apply() ставит animation = STATE_TO_CLIP[state],
#     flip_h = dir < 0, play(); offset/centered по §6; texture_filter по §2.9.
```

Требования:
- Переход состояния — мгновенный (`play(clip)` с начала, без доигрывания).
- `dir` меняет только `flip_h` (или `scale.x` знак), позицию узла не трогает.
- Не переопределять `self.modulate`.
- Анимация идёт по визуальным часам `AnimatedSprite2D` (speed_scale/`_process`) — не по
  физике; на паузе игры допустимо продолжать визуально (как сейчас).
- Работает без `Lemming`-родителя (меню/карта мира).

## 10. Критерии приёмки и автотесты (GUT)

Поставка принимается только при **зелёном полном сьюте** и зелёном верификаторе уровней:

```
godot --headless -s addons/gut/gut_cmdln.gd -gdir=res://tests -gexit
godot --headless --fixed-fps 2000 -s scripts/verify_levels.gd
```

Подрядчик добавляет `tests/test_lemming_sprite.gd` минимум со следующими проверками:

1. **API-паритет:** у `LemmingSprite` есть свойства `state:int` и `dir:int`; присвоение не падает.
2. **Полнота клипов:** `lemming_frames.tres` содержит все имена анимаций из §4; для каждого
   значения `Lemming.State` (0..12) `STATE_TO_CLIP` даёт существующий клип.
3. **Кадры есть:** в каждом клипе ≥ 1 кадр; у циклических `loop = true`, у `splat`/`cheer`
   `loop = false`.
4. **Якорь стопы:** при любом `state` нижняя граница видимого кадра приходится на локальный
   `y = 0 ± 1px` (через `offset`/`region` и размеры кадра — посчитать, не «на глаз»).
5. **Зеркало:** `dir = -1` ⇒ `flip_h == true` и `position` узла не изменился; `dir = 1` ⇒
   `flip_h == false`.
6. **Модулят не тронут:** после смены `state`/`dir` `self.modulate == Color(1,1,1,1)`
   (родитель сам управляет подсветкой).
7. **Без родителя:** инстанс `LemmingSprite` рисуется и анимируется как отдельный узел
   (добавить в дерево, прогнать кадр, проверить, что нет ошибок и анимация играет).
8. *(желательно)* **Детерминизм запекания:** прогон bake-инструмента дважды → одинаковый хэш
   выходных файлов.

Менять существующие тесты (`test_*`) и игровую логику ради прохождения **нельзя** — адаптер
обязан совпасть с уже зашитым контрактом. Если что-то в контракте мешает — это вопрос к нам,
а не повод править игру.

## 11. Производительность (мобильный бюджет)

- Целевой бюджет: **~100 анимированных леммингов в кадре @ 60 fps** на среднем телефоне
  (порядка iPhone SE / средний Android).
- **Один общий `SpriteFrames`** на всех (preload, одна текстура атласа в VRAM). Никаких
  `AnimationPlayer`/`Skeleton2D`/`SubViewport` на каждом юните в рантайме — скелет/риг живёт
  только в bake-инструменте.
- Атлас компактный; число кадров на клип — разумный минимум для читаемой анимации (ориентир
  6–12 кадров на цикл), не «киношная» плотность.
- Отрисовка — обычный `AnimatedSprite2D` (батчится). Без шейдеров на юнит без необходимости.

- Растровый арт «от руки» как исходник — исходник только векторный (SVG-риг).
